## 目标
1. 继续补齐尚未注解的 Handler（agents、knowledge、audit、workflow templates 等），让 `swag init` 能覆盖所有核心 API。
2. 修正工作流执行与工具接口的响应/注解瑕疵，并统一引用通用 DTO。
3. 配置并执行 Swagger 生成链路，再运行完整后端测试（含一次实际启动）验证。

## 实施步骤
1. **缺失注解补齐**
   - 针对 `backend/api/handlers/agents/*.go`、`handlers/knowledge/*.go`、`handlers/audit/*.go` 以及 `workflows/template_handler.go` 逐个路由添加 `@Summary/@Tags/@Security/@Param/@Success/@Failure/@Router`。
   - 引入 `response` 包，新增必要的请求/响应 DTO（例如 `AgentConfigListResponse`、`KnowledgeDocumentResponse`、`AuditLogListResponse` 等）并放入对应 handler 或新建 `dto.go` 文件，移除匿名结构体。
2. **工作流/工具修复**
   - `WorkflowExecuteHandler.ExecuteWorkflow`：修正被破坏的注释块，改用显式 `WorkflowExecutionResponse`（包含 execution_id/input/output/timestamps）并返回 202；保持 Swagger 注解与响应一致。
   - `ListExecutions`、`GetExecution` 等补充缺失的 `@Failure` 描述。
   - 工具 handler：确认分页/错误返回均使用 `response.ListResponse` 与 `response.ErrorResponse`；完善 `@Failure` 注解（如 403/500）。
3. **Swagger 生成链路**
   - 新建 `backend/cmd/server/doc.go`（或补充现有文件）写入 `//go:generate swag fmt` 与 `swag init -g cmd/server/main.go -o api/docs`，并补足顶层 `@title/@version/@host/@BasePath` 注释。
   - 更新 `backend/Makefile` 新增 `swagger` 目标（执行 `swag fmt && swag init ...`），确保 `tools/tools.go` 引入已安装的 swag CLI。
4. **生成与验证**
   - 执行 `cd backend && make swagger`，检查 `api/docs/swagger.json` / `.yaml` 变更，确保新增端点均出现。
   - 运行 `cd backend && go test ./...` 验证逻辑未受影响。
   - 实际启动一次后端：`cd backend && go run cmd/server/main.go`（或 `go build` + 运行），确认 Swagger 路由和主要 API 可正常起服。
5. **后续协同**（若时间允许）
   - 根据更新后的 Swagger 运行 `npm --prefix frontend run generate:api`，验证前端客户端生成完备（暂列为下一阶段，可在本轮交付后标注）。
